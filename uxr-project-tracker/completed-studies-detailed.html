FoundryIQ & Tools - Foundational Customer Interviews Irina Smoke 2026-04-14 SharePoint Doc Nine interviews with agent builders across enterprise and startup revealed that connecting data to agentic systems is the central pain point — developers struggle with fragmented data sources, inconsistent tooling, and unclear paths to grounding agents in trusted data. Foundry's opportunity lies in simplifying the data-to-agent pipeline, especially for knowledge retrieval and tool integration. Jobs in using data & tools with agents include: Finding and accessing data, Consolidating data from multiple sources, Normalizing and pre-processing messy, inconsistent data, Connecting to data sources and tools, Improving data retrieval (e.g., implement semantic search, build context graphs), Managing agent context effectively, Ensuring RBAC is respected, Keeping data up-to-date in production, and tracing agent behavior across data and tool calls.
Pain Points in Data & Tools:
  • Normalizing, “untangling,” connecting enterprise data (especially unstructured data) from a variety of sources and getting them ready for agent use is a time consuming, intensive process.
  • Expressing the structure and meaning of your data to agents is a challenge and customers do a variety of things, ranging from building knowledge graphs to using tools like Notion to write wiki-style articles for their agents to understand their datasets
  • Context engineering requires careful management, even with larger context windows
  • Customers are challenged with ensuring data security and privacy. For example, ensuring that RBAC is respected (i.e. if I query an agent it only shows me data I’m allowed to see) and data isolation between their end customers
  • Keeping RAG and graph indexes production fresh is a challenge - both using production data to improve their agents (a process with many manual steps today) and building data pipelines to ensure that new content is ingested properly in a timely manner.
Embr (Startup Cloud) UXR Collaboration with Azure Core Irina Smoke 2026-04-13 SharePoint Doc Interviews with 9 AI-native startup executives explored the "Day Zero" onboarding and neocloud-to-hyperscaler migration journey. Startups choose developer-first platforms like Vercel and Railway for speed, but hit scaling walls that force them toward Azure or AWS — the migration tipping point is compliance and GPU access, not feature gaps. Concept feedback on a dedicated "startup cloud" was cautiously positive, with participants wanting Azure's power without the enterprise complexity.
  • Consensus that core features of Embr aka Startup Cloud align with early-stage startup needs
  • Cost visibility and management was a major concern (especially with split billing when extending into Azure)
  • Language support for Go and Java requested (over .NET)
  • Opinionated defaults received, but transparency is crucial
  • Inflection points that necessitate a migration (Why do startups move from neocloud to hyperscaler?): Scalability and performance needs, Compliance and security requirements, Cost efficiency at scale, Customer requirement / GTM opportunities.
AITK Foundry Extensions Study Annemarie Fulcer 2026-04-13 None Agent Builder emerged as the highest-value feature and primary reason developers use the extension, but evaluation and tracing -- critical to the iterative build-test-refine loop -- are buried under Monitor instead of Build, breaking the workflow. Custom code training Foundry IA Annemarie Fulcer 2026-04-15 SharePoint Doc A survey of 27 developers plus live customer walkthroughs revealed strong preferences for how custom model training should be structured in Foundry. "Build" emerged as the clear winner for top-level navigation, while users insisted on keeping "Train" and "Fine-Tune" as distinct concepts — collapsing them causes real confusion. Buzzword-heavy labels like "AI" and "Custom" actively hurt clarity; precise terms like "Model fine-tuning" and "Task fine-tuning" are what developers actually want. AML custom code/model training capabilities ARE going to be in Foundry. Signals were proactively captured via survey, then validated with limited customers in live sessions given constraints.
Findings on Foundry IA for this scenario:
  • Build as a strong, intent driven entry point
  • The necessity of preserving train for full control workflows
  • The importance of fine tune vs train as task-dependent, not interchangeable
  • The functional preference toward intent/action oriented labeling.
Dev Workflow and Cloud Deployment Pilot Studies Taryn Bipat, Haley Welliver (RYLEM LLC) 2026-04-14 SharePoint Doc Five startup full-stack developers using Vercel, Netlify, Firebase, DigitalOcean, and Cloudflare were interviewed about how their app-building workflow is shifting from local IDE to cloud deployment toward a chat-first, browser-native paradigm. Every participant relied heavily on AI coding tools (GitHub Copilot, Claude Code, ChatGPT, Gemini, V0), suggesting the IDE-to-cloud transition is being accelerated by AI-assisted development. Pilot Study 1 (Pro-Devs in enterprise or larger startups)
  • AI is trusted for crafting code, not shipping it - Enterprise pro-devs actively use AI for code generation, research, and troubleshooting, but draw a firm boundary at production deployment. Automation without human review is seen as a liability.
  • Human-in-the-loop governance is non-negotiable - Developers fear audit failures and personal accountability if AI makes unchecked changes in production.
  • Deployment pain points: Visibility and Validation - The biggest time sink is not running pipelines, but understanding what failed, where, and why across fragmented tools (CDK, Terraform, Spinnaker, AWS Console, logs).
  • Trust in AI is driven by transparency and granular control - Developers want AI outputs that are explainable, incremental, and reviewable. Including source links, diffs, and per-command approvals. Large changes dramatically reduce trust.
Pilot Study 2 (Pro-Devs or hobbyists in smaller, fast-paced startup cultures)
  • Lightweight platforms win because they reduce cognitive load - Platforms like Vercel and Netlify succeed by offering faster previews, opinionated defaults, and transparent pricing — while hyperscalers feel powerful but overwhelming.
  • Pro-devs want automated deployments ...but need the ability to review, approve, and manually control production-impacting changes when necessary.
  • Cost & security are gating factors to AI-assisted deployments - Unexpected billing, pricing volatility, and required credit cards are major deterrents (even when tools are technically a better option.) Participant hesitation in having AI handle API keys and .env files.
Potential UI ideas for an AI-assisted deployment product
  • Lightweight validation over heavyweight dashboards - A visual to-do list or deployment timeline showing completed and pending steps (some completed by dev and some completed by AI) is preferred over complex monitoring views.
  • Continuous, connected workflow - Support seamless movement between IDE, terminal, and cloud portal, avoiding context-switching while allowing deeper inspection when needed (perhaps via AI provided links)
  • Trust cues built into the interface - Use checklists, command logs, diffs, security indicators, and provenance (“AI did this” vs. “you did this”) to reinforce trust and confidence.
