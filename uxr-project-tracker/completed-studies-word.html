Knowledge, Tools, and Agent Building - JTBD and Pain Points - Customer Interview Learnings Irina Smoke Jobs in using data & tools with agents include: Finding and accessing data, Consolidating data from multiple sources, Normalizing and pre-processing messy, inconsistent data, Connecting to data sources and tools, Improving data retrieval (e.g., implement semantic search, build context graphs), Managing agent context effectively, Ensuring RBAC is respected, Keeping data up-to-date in production, and tracing agent behavior across data and tool calls.
Pain Points in Data & Tools:
  • Normalizing, “untangling,” connecting enterprise data (especially unstructured data) from a variety of sources and getting them ready for agent use is a time consuming, intensive process.
  • Expressing the structure and meaning of your data to agents is a challenge and customers do a variety of things, ranging from building knowledge graphs to using tools like Notion to write wiki-style articles for their agents to understand their datasets
  • Context engineering requires careful management, even with larger context windows
  • Customers are challenged with ensuring data security and privacy. For example, ensuring that RBAC is respected (i.e. if I query an agent it only shows me data I’m allowed to see) and data isolation between their end customers
  • Keeping RAG and graph indexes production fresh is a challenge - both using production data to improve their agents (a process with many manual steps today) and building data pipelines to ensure that new content is ingested properly in a timely manner.

Dev Workflow & Startup Cloud Deployment - Pilot 2 Taryn Bipat, Haley Welliver (RYLEM LLC) Pilot Study 1 (Pro-Devs in enterprise or larger startups)
  • AI is trusted for crafting code, not shipping it - Enterprise pro-devs actively use AI for code generation, research, and troubleshooting, but draw a firm boundary at production deployment. Automation without human review is seen as a liability.
  • Human-in-the-loop governance is non-negotiable - Developers fear audit failures and personal accountability if AI makes unchecked changes in production.
  • Deployment pain points: Visibility and Validation - The biggest time sink is not running pipelines, but understanding what failed, where, and why across fragmented tools (CDK, Terraform, Spinnaker, AWS Console, logs).
  • Trust in AI is driven by transparency and granular control - Developers want AI outputs that are explainable, incremental, and reviewable. Including source links, diffs, and per-command approvals. Large changes dramatically reduce trust.
Pilot Study 2 (Pro-Devs or hobbyists in smaller, fast-paced startup cultures)
  • Lightweight platforms win because they reduce cognitive load - Platforms like Vercel and Netlify succeed by offering faster previews, opinionated defaults, and transparent pricing — while hyperscalers feel powerful but overwhelming.
  • Pro-devs want automated deployments ...but need the ability to review, approve, and manually control production-impacting changes when necessary.
  • Cost & security are gating factors to AI-assisted deployments - Unexpected billing, pricing volatility, and required credit cards are major deterrents (even when tools are technically a better option.) Participant hesitation in having AI handle API keys and .env files.
Potential UI ideas for an AI-assisted deployment product
  • Lightweight validation over heavyweight dashboards - A visual to-do list or deployment timeline showing completed and pending steps (some completed by dev and some completed by AI) is preferred over complex monitoring views.
  • Continuous, connected workflow - Support seamless movement between IDE, terminal, and cloud portal, avoiding context-switching while allowing deeper inspection when needed (perhaps via AI provided links)
  • Trust cues built into the interface - Use checklists, command logs, diffs, security indicators, and provenance (“AI did this” vs. “you did this”) to reinforce trust and confidence.
